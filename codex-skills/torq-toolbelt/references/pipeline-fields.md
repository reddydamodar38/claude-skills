# Pipeline Fields

Use this file when the user asks what a TORQ field means or needs help filling out `Create Pipeline` or the generated testing pipeline.

## Two Different Jobs

Distinguish between:

- `Create Pipeline`
  - creates or updates the testing pipeline
- generated testing pipeline
  - the job the operator actually runs for a test

Do not mix these field sets together.

## `Create Pipeline` Fields

These define the testing pipeline that TORQ will create.

- `organization`
  - top-level folder grouping on the share
- `subproject`
  - the specific test project or testing pipeline name
- `test_type`
  - broad testing family such as Alva or EJS
- `automation`
  - selected automation family such as Gatling or Eggplant
- `automation_config`
  - technology-specific inputs used to seed parts of the created pipeline config
- `tag_name`
  - AutoCatalog-related field; do not recommend by default because AutoCatalog is deprecated
- `workflows`
  - AutoCatalog-related field; same caution as above
- `email`
  - default recipients for the generated testing pipeline
- `environment_name`
  - short Millennium environment name
- `dns_suffix`
  - DNS suffix for the environment
- `timezone`
  - TORQ-side timezone used by the generated pipeline
- `load_generator_config`
  - seeded YAML for the created testing pipeline
- `test_config`
  - seeded YAML for the created testing pipeline

## Generated Testing Pipeline Fields

These are the fields operators use when they actually run a test.

- `description`
  - free-text explanation of the run
- `suffix`
  - appended to the generated test name
- `user_count`
  - requested user or load count
- `duration`
  - run duration choice generated by `Create Pipeline`
- `flashback`
  - whether to flash back the database before prep
- `restore_point`
  - flashback target name
- `load_generator`
  - selected load generator types for the run
- `load_generator_config`
  - YAML that controls load generator behavior
- `test_config`
  - YAML that controls TORQ and environment behavior
- `pause_before_submit`
  - pauses after prep and before submit
- `test_plan`
  - currently created as `Adhoc` or `Scalability`
- `email`
  - comma-separated notification list
- `custom_variables`
  - free-form variables in `key=value` format

## Current Field Notes

- `duration`
  - `Create Pipeline` seeds the available values based on selected test type
- `load_generator`
  - the selected type gates which parts of `load_generator_config` will actually run
- `pause_before_submit`
  - intentional operator pause, not necessarily a failure
- `test_plan`
  - `Scalability` is not just a label; it can trigger additional generated runs

## Important Drift Warning

Older docs may mention:

- `custom_scripts`
  - current generated testing pipelines do not expose this as a separate field
  - use `test_config` instead

Older docs may also understate `test_config` as mostly Alva-related.
Current library code supports broader usage than that.

## Best Source for Exact Current Field Shape

When the docs and the live job differ, trust:

- `torq-image/casc_configs/create-pipeline.yaml`

That file defines both the `Create Pipeline` fields and the generated testing pipeline fields.
